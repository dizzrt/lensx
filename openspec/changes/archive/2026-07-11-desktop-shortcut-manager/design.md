## Context

当前 Rust 侧已经有托盘和全局快捷键的基础实现：应用启动时注册一个固定的 `Ctrl+Shift+Space`，触发后直接 `show()` 主窗口；窗口关闭事件被拦截并转为 `hide()`，使进程保持运行。这个实现足够验证“隐藏后快捷键唤起”的最小闭环，但快捷键、动作和窗口生命周期耦合在同一个回调里，后续扩展多个快捷键或用户自定义快捷键时会快速失控。

本变更只处理 Rust/Tauri 侧底座。Vue 设置页、快捷键录制 UI、用户偏好持久化 UI 都后置。设计目标是让当前默认快捷键也走未来可复用的注册表和动作路由，避免先写死一版、再重构一版。

## Goals / Non-Goals

**Goals:**

- 在 Rust 侧建立快捷键绑定模型，至少包含绑定 ID、快捷键组合、动作 ID、启用状态。
- 将默认 `Ctrl+Shift+Space` 从硬编码窗口操作迁移为默认绑定。
- 提供快捷键注册、注销、重新注册和冲突检测的内部管理边界。
- 提供启动器窗口动作路由，覆盖 `launcher.show`、`launcher.hide`、`launcher.toggle`。
- 改善快捷键唤起行为：显示时恢复窗口、显示窗口并请求焦点；隐藏时保持进程常驻。
- 保留未来暴露给 Vue 设置页的稳定 Rust API 形状，但本次不实现设置页。

**Non-Goals:**

- 不实现 Vue 快捷键设置页、快捷键录制控件或用户可见配置文案。
- 不实现用户配置持久化；默认绑定可以先在 Rust 内部静态声明。
- 不实现插件、命令注册表或命令级快捷键。
- 不实现开机自启动。
- 不改变启动器面板布局、搜索逻辑或内容数据来源。

## Decisions

### 1. 使用数据化绑定替代回调内硬编码

定义 Rust 侧 `ShortcutBinding` 与 `ShortcutAction`，让快捷键注册和动作执行分离：

```text
ShortcutBinding
├─ id: "launcher.toggle"
├─ accelerator: Ctrl+Shift+Space
├─ action: LauncherToggle
└─ enabled: true

Global shortcut callback
        │
        ▼
ShortcutManager 根据已注册快捷键找到 action
        │
        ▼
ActionRouter 执行 launcher.toggle
```

理由：新增快捷键时只增加绑定和动作映射，不需要继续在插件 handler 里追加特殊判断。

替代方案：继续在 `shortcut.rs` 中用多个 `if shortcut == ...` 分支。这个方案初期代码更少，但不支持低成本动态重载和冲突处理。

### 2. ShortcutManager 拥有注册生命周期

`ShortcutManager` 负责：

- 保存当前已启用绑定与注册结果。
- 启动时注册默认绑定。
- 重新注册时先注销旧绑定，再注册新绑定。
- 调用 `is_registered` 或等价能力做冲突检测，并把失败返回为 typed error。

理由：生命周期集中后，未来设置页只需要调用“应用一组绑定”，不需要理解 Tauri 插件细节。

替代方案：把注册逻辑散落在 Tauri command 中。这个方案会让命令层变厚，违背“命令薄、系统行为在 Rust 服务中”的项目约束。

### 3. 动作路由先只覆盖启动器窗口

本次动作集合只包含：

- `launcher.show`
- `launcher.hide`
- `launcher.toggle`

`launcher.toggle` 应根据主窗口当前可见状态决定显示或隐藏；显示路径应尽量执行 `unminimize`、`show`、`set_focus`。如果某个平台或窗口状态不支持其中某一步，应返回错误或记录失败，不应直接 `unwrap()` 导致进程崩溃。

理由：这覆盖当前需求，并为未来命令动作保留扩展点，但不提前引入尚未实现的命令系统。

替代方案：直接把动作定义为字符串并动态分发。这个方案灵活，但第一版缺少编译期约束，容易把未实现能力误描述为可用能力。

### 4. 前端 API 只预留边界，不做 UI

如本次需要暴露命令，应保持最小、稳定、 typed 的接口，例如：

- 查询默认或当前绑定。
- 应用一组绑定。
- 重置绑定。

字段使用 `snake_case`。如果实现中没有前端调用需求，可以先保持内部 Rust API，不新增 Tauri command。

理由：用户已明确 Vue 侧不着急；过早做设置页 API 可能会固化还没验证的前端交互模型。

替代方案：一次性做完整前端设置页。这个方案范围更大，会引入 i18n、主题、表单校验和快捷键录制交互，不适合当前阶段。

### 5. 用户自定义快捷键持久化长期归入统一偏好设置服务

本次不实现用户自定义快捷键持久化；后续需要支持设置页修改快捷键时，持久化所有权应归入统一偏好设置服务，而不是直接让 Vue 设置页或 Tauri store 成为快捷键配置的事实源。

理由：快捷键属于桌面系统行为，注册、冲突检测、恢复默认和平台差异都在 Rust 侧。统一偏好设置服务可以作为主题、语言、窗口行为、快捷键和未来扩展配置的共同入口，同时保持 Vue 只通过 typed Tauri command 读写偏好。

替代方案：

- 使用 Tauri store：接入快，但配置所有权容易偏向前端，后续复杂设置会分散。
- 使用 Rust 本地配置文件：适合短期落地，但如果未来设置项增加，仍需要演进为统一偏好入口。

## Risks / Trade-offs

- [Risk] 不做持久化会让“自定义快捷键”暂时不可用。→ Mitigation: 本次只承诺 Rust 侧可重载边界，后续持久化和 Vue UI 作为独立变更接入。
- [Risk] 跨平台快捷键表示和冲突行为不完全一致。→ Mitigation: 先用 `tauri-plugin-global-shortcut` 的类型作为内部注册源，错误向上传递，避免吞掉失败。
- [Risk] `toggle` 在部分平台上窗口可见性判断可能与最小化、失焦状态不完全等价。→ Mitigation: 动作实现中区分 show/hide/focus 步骤，并保留可测试的行为边界。
- [Risk] 如果提前引入字符串动作 ID，未来命令系统接入可能出现命名不一致。→ Mitigation: Rust 内部优先使用 enum，只有跨前端边界时再序列化为稳定字符串。

## Migration Plan

1. 新增快捷键模型、默认绑定、管理器和启动器窗口动作模块。
2. 将现有固定快捷键注册逻辑迁移到 `ShortcutManager`。
3. 将默认快捷键触发行为从直接 `show()` 改为动作路由执行 `launcher.toggle`。
4. 保留关闭窗口时隐藏的常驻行为，并移除窗口操作中的 `unwrap()`。
5. 编译验证 Rust/Tauri 侧，必要时运行前端构建确保 Tauri 集成未破坏。

回滚策略：如果新管理器导致快捷键注册失败，可以回退到原有单快捷键注册模块；由于本变更不迁移用户数据，回滚不涉及数据兼容。
