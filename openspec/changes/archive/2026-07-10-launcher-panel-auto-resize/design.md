## Context

当前前端入口 `App.vue` 只渲染一个基础 `n-auto-complete` 输入框和左侧拖拽区域；Tauri 主窗口在配置中固定为 `800x60`、不可手动缩放。新的启动器面板宽度固定为 `650px`，最大窗口高度固定为 `800px`，需要承载默认状态与搜索状态的内容主体，因此仅做 CSS 自适应不足以完成体验，原生窗口高度也必须随内容变化。

本次变更仍属于表现层和窗口壳行为，不引入真实搜索、最近使用记录、固定项持久化、扩展市场或账号系统。Vue 负责视图状态、mock 数据、内容高度测量和调用 Tauri 窗口 API；Tauri/Rust 侧继续负责窗口生命周期、快捷键、托盘和关闭隐藏等已有桌面行为。

## Goals / Non-Goals

**Goals:**

- 建立一体化 launcher 面板：顶部搜索输入区，下方内容主体。
- 默认状态展示“最近使用”和“已固定”；搜索状态仅展示“匹配结果”。
- 搜索区和主体区在视觉上属于同一张卡片，没有明显分隔痕迹。
- 主体高度随内容变化，并在超过最大高度后改为主体内部滚动。
- 原生主窗口高度跟随面板内容自动调整，采用顶部固定、向下展开或收起策略。
- 输入框可点击、可输入；输入框外的可拖拽区域可移动窗口。
- 新增文案接入应用 i18n，样式遵循 Naive UI token、UnoCSS 和 Less 分工，并兼容明暗主题。

**Non-Goals:**

- 不实现真实搜索索引、排序、命令注册或扩展运行时。
- 不实现最近使用和已固定的真实数据读取、写入或持久化。
- 不展示市场精选、头像栏或账号入口。
- 不调整全局快捷键、托盘菜单、关闭隐藏、始终置顶等桌面生命周期策略。
- 不引入新的 UI 或状态管理依赖。

## Decisions

### 1. 使用单一 launcher card 承载搜索区和主体区

搜索区与主体区共用一个根卡片容器，由卡片提供背景、圆角、阴影、边框和主题变量。内部通过间距、字号、透明度和分区标题表达层级，不使用强分割线。

备选方案是将搜索栏和主体做成两个独立容器再靠边距贴合。该方案容易产生视觉断层，也会让窗口高度测量和滚动边界更复杂，因此不采用。

### 2. 使用 Vue 本地状态表达表现层

前端维护 `query`、默认分区 mock 数据和搜索匹配 mock 数据。`query` 为空时展示默认面板；`query` 非空时展示匹配结果面板。

备选方案是预先设计真实搜索接口或 Rust command。本次目标是表现层和窗口尺寸行为，过早引入搜索数据契约会扩大范围，因此不采用。

### 3. 使用 ResizeObserver 驱动窗口高度

前端通过 `ResizeObserver` 观察 launcher card 的实际高度，在高度变化后计算目标窗口高度，并调用 Tauri window API 调整当前窗口尺寸。窗口宽度固定为 `650px`；高度计算需要包含最小高度、`800px` 最大窗口高度和舍入策略，避免输入过程中频繁抖动。

推荐数据流：

```text
query / 内容变化
      │
      ▼
Vue 渲染 launcher card
      │
      ▼
ResizeObserver 读取 card 高度
      │
      ▼
clamp(min_window_height, measured_height, 800)
      │
      ▼
getCurrentWindow().setSize(new LogicalSize(650, height))
```

备选方案是在 Rust 侧主动计算窗口高度。Rust 无法直接掌握前端 DOM 内容高度，需要额外事件和契约，复杂度更高，因此本次由 Vue 测量并请求窗口调整。

### 4. 采用顶部固定、向下展开策略

窗口高度变化时只调整 size，不同步调整 position。这样窗口左上角保持不变，搜索栏位置稳定，内容向下展开或收起。

备选方案是保持窗口中心点不变，即每次 resize 后重新计算并设置 position。该方案视觉上可能更居中，但输入时窗口会产生纵向位移，影响焦点和视线稳定，因此不采用。

### 5. 主体内部滚动，而不是整窗滚动

launcher card 外层不承担滚动；搜索区保持稳定，内容主体设置最大高度和 `overflow-y: auto`。当内容超过主体最大高度时，原生窗口停止增高，滚动条只出现在主体内部。

备选方案是让整个页面滚动。该方案会让搜索框跟着滚动，违背 launcher 高频输入体验，因此不采用。

### 6. 拖拽区域避开真实输入控件

卡片和搜索区背景可以标记 `data-tauri-drag-region`，但输入控件本身不得作为拖拽区域。这样用户可以点击输入、选择文本，也可以拖拽输入框之外的空白区域移动窗口。

如果使用 `data-tauri-drag-region` 覆盖输入控件，可能破坏光标定位、文本选择和输入行为，因此必须避免。

### 7. 延续现有前端基座

实现应继续使用当前 `AppProviders` 提供的 Naive UI provider、全局样式、主题和 locale/dateLocale 同步能力。新增用户文案写入应用 i18n messages；静态布局优先用 UnoCSS，复杂卡片、滚动条、状态样式和主题绑定写在 Less 中。

## Risks / Trade-offs

- [风险] ResizeObserver 与 `setSize` 互相触发可能造成重复 resize 或轻微抖动 → 记录上一次应用的高度，只有高度变化超过阈值时才调用 `setSize`。
- [风险] Tauri v2 权限不足导致前端窗口 API 调用失败 → 实现时检查 capabilities，为主窗口显式授予所需 window 权限，并保持权限最小化。
- [风险] 不同平台窗口边框、缩放因子和逻辑像素表现不同 → 使用 `LogicalSize`，并通过 `pnpm run tauri dev` 在目标平台做烟测。
- [风险] `800px` 最大窗口高度在较小屏幕上可能遮挡屏幕区域 → 本次按已确认固定值实现，后续如需适配不同屏幕尺寸再单独设计屏幕约束策略。
- [风险] mock 数据可能被误解为真实能力 → UI 文案和任务范围明确仅为表现层，不接入持久化或真实搜索。

## Migration Plan

1. 在前端替换现有单行启动器界面为一体化面板。
2. 添加应用 i18n 文案和 mock 数据展示。
3. 添加 ResizeObserver 和 Tauri window resize 调用。
4. 调整 Tauri capabilities 中与窗口拖拽、窗口尺寸调整相关的最小权限。
5. 使用构建和 Tauri dev 验证布局、输入、拖拽、窗口高度变化和内部滚动。

如果实现后窗口 resize 行为不稳定，可临时保留新面板视觉布局，并将窗口高度调整回固定高度作为回滚路径；该回滚不影响搜索数据或持久化能力，因为本次不引入这些能力。
